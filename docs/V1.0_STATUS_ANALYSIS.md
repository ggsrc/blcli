# blcli v1.0 能力实现状态分析

> 分析日期：2026-01-27  
> 基于：`docs/Roadmap.md` v1.0 规划 vs 当前代码实现

## 执行摘要

**总体完成度：约 75%**

- ✅ **已完成**：init 基础功能、apply 命令（含依赖排序与执行计划、--dry-run）、check 命令、**status 命令**、**rollback 命令**（独立 `blcli rollback`）、**进度显示与持久化**（init/apply all）、模板加载、apply init-repos
- ⚠️ **部分实现**：init 增强（有依赖与进度，无并行；Git 提交需手动 `apply init-repos`）；回滚为 config 驱动、无检查点机制
- ❌ **未实现**：进度中断后恢复（Resume）、apply kubernetes/gitops 按 project、完善错误提示、操作历史/审计  
- **不要求**：并行初始化、自动提交 Git、失败自动重试、多模板源合并、模板版本管理

### 1.4 增强模板系统

| 能力 | Roadmap 要求 | 当前状态 | 说明 |
|------|-------------|---------|------|
| **模板版本管理** | 不要求 | — | 不增加版本锁定/更新检测；当前支持 @branch 即可 |
| **多模板源合并** | 不要求 | — | 不支持、也不计划：一次操作只从一个模板仓库加载 |
| **单次操作单仓库** | ✅ 要求 | ✅ **已实现** | 每次通过 `--template-repo` 指定一个仓库；不同命令/不同时机可用不同仓库 |

**当前实现与约定**：
- ✅ 支持 GitHub 仓库（公开/私有）、本地路径、缓存、`--force-update`
- ✅ 支持 `@branch` 形式（如 `github.com/user/repo@main`）
- **不支持**：一次加载合并多个模板仓库（无此需求）；不增加模板版本管理
- **“多模板源”说明**：若指“不同时机用不同仓库”，已支持（每次传入不同 `--template-repo`）；若指“一次 init 从多个仓库合并模板”，不做

**实现位置**：`pkg/template/config.go:ResolveKubernetesDependencies`

**实现方式**：
- ✅ 使用深度优先搜索（DFS）进行拓扑排序
- ✅ 循环依赖检测（visiting 标记）
- ✅ 在 `apply kubernetes` 中调用（`pkg/bootstrap/apply_kubernetes.go:183`）

**示例**：
```go
orderedComponents, err := kubernetesConfig.ResolveKubernetesDependencies(allComponentNames)
// 然后按 orderedComponents 顺序执行
```

### ✅ Terraform Apply - 已实现依赖排序

**当前实现**：
- ✅ `pkg/template/config.go` 中已实现 `ResolveTerraformDependencies`（DFS 拓扑排序）
- ✅ `pkg/bootstrap/apply_terraform.go` 在提供 template repo 时加载 config 并按依赖顺序执行项目
- Init 目录仍按数字前缀排序，项目目录按依赖排序

### ✅ 执行计划输出 - 已实现

## 能力变更说明

### ✅ 已实现但命名/方式不同

1. **`install` → `apply`**
   - Roadmap 规划 `install`，实际实现 `apply`
   - 功能基本对齐，但缺少分批安装、回滚、状态持久化

2. **Git 提交功能**
   - Roadmap 要求自动提交，实际实现为 `apply init-repos` 子命令
   - 需要用户手动调用，不是自动的

### ⚠️ 部分实现

1. **依赖管理**
   - Terraform/Kubernetes apply 均已按 config 依赖排序
   - 无依赖图可视化、无单独“依赖检查”子命令

2. **回滚**
   - 独立 `blcli rollback` 已实现（按 config 的 Rollback 配置），无检查点机制、非 apply 失败自动触发

### ❌ 完全未实现

1. **进度中断后恢复（Resume）** - 有 LoadProgressTracker，入口未接恢复流程
2. **apply 按 project** - Kubernetes/GitOps 待支持 `--project`
3. **完善错误提示** - 错误信息需更清晰、可操作（不添加重试）
4. **操作历史/审计** - 无独立操作历史
5. **不要求**：并行初始化、自动提交 Git、失败重试、多模板源合并、模板版本管理
